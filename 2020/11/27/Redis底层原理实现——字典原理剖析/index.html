Redis底层原理实现——字典原理剖析

发表于2020-11-27|更新于2020-11-27|Redis原理篇
|字数总计:1.8k|阅读时长:6分钟|阅读量:|

字典

  字典，一个键 key 和一个值 value 进行关联，一对一的关系映射，这些关联又称为键值对。

  在 Redis 中的应用十分广泛，比如 Redis 的数据库就是使用字典作为底层实现的，对数据库的增、删、改、查操作就是构建在对字典的操作之上的。字典还是哈希键的底层实现之一，当一个哈希键包含的键值对比较多，又或者键值对中的元素都是比较长的元素时，Redsi 就会使用字典作为哈希键的底层实现。

  • ziplist（压缩列表）：当哈希类型元素个数 hash-max-ziplist-entries 小于配置（默认512），同时所有值都小于 hash-max-ziplist-value 配置（默认64字节）时，Redis 会使用 ziplist 作为 hash 的内部实现。
  • hashtable（哈希表）：当无法满足 ziplist 的条件时，会使用 hashtable 作为底层实现，读写效率为 O(1)。

字典的内部实现

哈希表

  Redis 字典所使用的哈希表结构体，定义如下：

  • table：是一个数组，数组中的每一个元素均指向字典节点 dictEntry 结构体，每一个 dictEntry 均是一个键值对，Redis 采用拉链法来解决hash冲突，当多个元素hash值相同时，那么会形成一个链表结构存储。
  • size：表示当前哈希数组的大小。
  • sizemask：用来计算索引值，总是等于 size - 1。idx = hash & sizemask
  • used：用来表示当前哈希表中已有节点的个数。

哈希表节点

  • key：用来表示哈希表节点中的键，可以是任意类型的。
  • next：用来指向下一个哈希表节点的指针，如果出现哈希冲突那么将会用拉链法解法。
  • value：节点中值，可以是任意类型，也可以是整数。

字典

  哈希表由两个组成，ht[0] 作为平时增删改查操作的对象，而 ht[1] 作为扩容的时候选择，哈希表在扩容的时候采用了一种渐进式哈希扩容，不是一次性将所有键值对都移植到新的哈希表上，如果原有的键值对数量比较多，那么Redis将会被阻塞一段时间。rehash，表示当前正在进行的移植的数组索引，如果没有进行扩容操作，那么这个值为 -1。

rehash

  哈希表的收缩和扩展根据一个负载因子（loadfactor），当哈希表保存的键值对数量太多或者太少时，需要对哈希表进行扩展和收缩。

扩展&收缩步骤

  1. 为 ht[1] 哈希表分配空间，这个哈希表的大小取决于要执行的操作，以及当前 ht[0] 中键值对的数量。
    1. 如果执行的扩展操作，那么 ht[1] 的大小为第一个大于等于 ht[0].used * 2 的 2^n。
    2. 如果执行的是收缩操作，那么ht[1] 的大小为第一个小于等于 ht[0].used 的 2^n。
  2. 将保存在 ht[0] 上面的所有键值对进行 rehash 到 ht[1] 上面，rehash 指的是重新计算哈希值和索引值
  3. 当所有 ht[0] 上面的元素都迁移到了 ht[1] 上之后，ht[0] 会被释放，然后 ht[1] 设置为 ht[0]，并在 ht[1] 新建一个空白哈希表，为了下一次的rehash做准备。

渐进式 rehash

  rehash过程，并不是一次性的，而是分多次、渐进式地完成的，这样做的好处有，当键值对个数比较多时，可以防止在rehash过程中，服务器处于阻塞状态。

  rehash过程如下：

  1. 为ht[1]分配空间,让字典同时持有ht[0]和ht[1]两个哈希表。
  2. 在字典中维持一个索引计数器变量 rehashidx,并将它的值设置为0,表示 rehash工作正式开始。
  3. 在 rehash进行期间,每次对字典执行添加、删除、查找或者更新操作时,程序除了执行指定的操作以外,还会顺带将ht[0]哈希表在 rehashidx索引上的所有键值对rehash到ht[1],当 rehash工作完成之后,程序将 rehashidx属性的值增一。
  4. 随着字典操作的不断执行,最终在某个时间点上,ht[01的所有键值对都会被hash至ht[1],这时程序将 rehashidx属性的值设为-1,表示 rehash操作已完成渐进式 rehash的好处在于它呆取分而治之的方式,将 rehash键值对所需的计算工作均摊到对字典的每个添加、删除、查找和更新操作上,从而避免了集中式 rehash而带来的庞大计算量。

  rehashidx，记录的是当前正在进行的rehash的索引数组下标值，如果没有进行的rehash操作，那么这个值为 -1。

  在rehash过程中，字典会同时使用 ht[0] 和 ht[1] 两个哈希表，所以在 rehash 期间内，字典的增、删、改、查操作会在两个哈希表上进行。在 rehash 过程中 ht[0] 这个哈希表不会在发生新增操作，这样就保证了 ht[0] 的键值对只会减少不会增加。

总结

  • 字典结构被广泛应用于Redis中的各种功能，包括数据库和哈希键。
  • Redis中使用哈希表作为底层实现，每个字典带有两个哈希表，一个平时使用，一个仅在进行rehash使用。
  • 哈希表采用链地址法来解决键冲突，被分配到同一个索引上的多个键值对会连接成一个单向链表。
  • 在对哈希表进行扩展和收缩时，这个过程不是一次性进行完毕的，是分多次的，渐进式的完成rehash操作，当整个操作结束时，将 ht[0] 这个哈希表指向 ht[1]，最后释放掉 ht[1] 。
